Introduction

OOP is objectively better.

  • Prettier
  • Easier to maintain - better separation and documentation
  • Easier to test
  • IDE helps understand the code
Andreas

But old-school programming was so much easier…

  • Yeah….
  • Sure but so are one click setup options

Then why Commerce 2.x is a complete rewrite?

  • All the OOP goodness
  • Changes to the architecture
  • Simplification

Migration

Commerce 1.x to Commerce 2.x

Reasons

  • Drupal 8, duh…
  • !Sunk cost
  • Less contrib code
  • Less custom code
Andreas

How

  • Migrate not upgrade.
  • Nodes -> products, products -> product variations.
  • New Attributes (no more taxonomies?)
  • No UI modules - installing Product provides entity type AND the interface
  • No Rules - ANARCHY?
  • Price components vs adjustments
  • No more hooks (or at least not that many) - we will cover that more in next section
Jakub

Maintainability

  • Backward compatibility (sometimes broken - find examples for payments and price formatter)
  • SOLID - good architecture requires principles
  • composer update drupal/commerce --with-dependencies
Andreas

Drupal 7 hook goodness

  • hook_commerce_payment_method_info (+bunch of callbacks)
  • hook_commerce_payment_transaction_status_info
  • hook_commerce_checkout_complete
  • hook_commerce_checkout_page_info
  • hook_commerce_cart_line_item_refresh/hook_commerce_cart_order_refresh
  • hook_commerce_cart_product_comparison_properties_alter
